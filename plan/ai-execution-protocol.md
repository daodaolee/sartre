# AI 执行协议（含 Definition of Done）

> **强制约束，最高优先级。** 本文件定义 AI 执行者如何消费规范、如何工作、遇到问题如何判断与修正、何时必须升级给人。核心诉求是"AI 检查健壮性可维护性、遇到问题进行判断和修正"——本文件是这套自治+自纠机制的规则书。
>
> 本文件合并了原 plan/06 的三级 Definition of Done（见第 10 节）。与 [workflow/plan-ledger-sop](../workflow/plan-ledger-sop.md)（跨会话恢复）、[workflow/WORKFLOW](../workflow/WORKFLOW.md)（OpenSpec 流程）配合。

## 1. AI 执行者的角色与边界

```
┌─────────────────────────────────────────────────────────────┐
│  实施 AI (Implementer)                                       │
│   - 读 plan → 写测试 → 写实现 → 自审 → 出审计                │
│   - 在授权范围内自主决策与修正                               │
│   - 越界则升级给人                                           │
├─────────────────────────────────────────────────────────────┤
│  复审 AI (Reviewer，关键模块)                                │
│   - 独立对照约束层找违规，不信任实施 AI 的自述               │
├─────────────────────────────────────────────────────────────┤
│  人 (Human)                                                  │
│   - 裁决升级事项、修订约束(ADR)、确认里程碑                  │
└─────────────────────────────────────────────────────────────┘
```

## 2. 标准工作循环

每开始一个模块/任务，AI 严格按此循环：

```
┌─────────────────────────────────────────────────────────────┐
│                    单模块标准工作循环                        │
└─────────────────────────────────────────────────────────────┘

 [加载上下文]
   按 AGENTS.md 必读顺序读：本协议 → spec/(宪法) → workflow/ → 00-master-plan → 模块plan
   │
   ▼
 [确认就绪]
   依赖模块是否已 ✅？否 → 不能开始，回看板找可做的
   理解模块契约与验收信号？否 → 先澄清(见第6节)
   │
   ▼
 [写测试] ◀─── 先写！测试表达契约与必测场景(spec/TestStrategy)
   │
   ▼
 [写实现] ── 红→绿，遵守 spec/ 全部宪法
   │
   ▼
 [自动化门禁] ── harness:regression + drift（编译/测试/lint/Token/循环/密钥）
   │      │
   │   不过 → 修复 → 重跑
   ▼
 [自审] ── 逐条对照第10节 DoD + 各宪法自检清单，记录证据
   │
   ▼
 [出审计报告] ── templates/audit-report，每条带证据+证据等级
   │
   ▼
 [复审(关键模块)] ── 独立 reviewer agent
   │
   ▼
 [更新留痕] ── 模块plan定稿 + master-plan看板 + PLAN_LEDGER + 必要的ADR
   │
   ▼
 [完成] ── 仅当 DoD 全满足
```

## 3. 遇到问题的判断框架（核心）

AI 实施中遇到的问题分四类，每类有明确处置。**判断的第一步永远是分类。**

```
┌─────────────────────────────────────────────────────────────┐
│              遇到问题 → 先分类 → 按类处置                    │
└─────────────────────────────────────────────────────────────┘

  遇到问题/卡住
       │
       ▼
  ┌─────────────────────────────────────┐
  │ 这是哪类问题？                       │
  └──┬──────┬──────────┬──────────┬──────┘
     ▼      ▼          ▼          ▼
  [A实现]  [B设计缺陷]  [C约束冲突]  [D范围/需求]
  局部技术  plan设计    约束本身     超出plan
  问题      有错        矛盾/过严     或需求模糊
     │      │          │          │
     ▼      ▼          ▼          ▼
  自主修复  提案修正    必须升级    必须升级
  (第4节)   plan(第5节)  (第6节)    (第6节)
```

### A 类 — 实现问题（自主解决）

局部技术问题：编译错误、测试不过、某个算法实现、库 API 用法。

**处置**：AI 自主解决，不需升级。但遵守"失败循环识别"——

```
同一问题尝试 2 次仍失败？
   │
   ├─ 否 → 继续尝试
   └─ 是 → 停止打补丁，诊断根因
            │
            ├─ 找到根因 → 换根本性方案
            └─ 根因指向 B/C/D 类 → 转对应流程
```

⚠️ 禁止：反复对同一错误做微调式打补丁（反模式，见全局规则）。两次失败必须升维到根因分析。

### B 类 — 设计缺陷（提案修正 plan）

实施中发现 plan/modules 的设计有错或不可行（如某接口设计导致循环依赖、某状态机漏了状态）。

**处置**：
1. 停止按错误设计实现
2. 诊断：错在哪、为什么、影响哪些模块
3. 形成**修正提案**：写清问题 + 建议改法 + 影响面
4. 若仅影响本模块 plan 且不违反约束层 → AI 可自主修订模块 plan，**留痕**（在模块 plan 的"变更记录"区记录）
5. 若影响契约（05）或其他模块 → 升级（第6节）

### C 类 — 约束冲突（必须升级）

发现约束层（01/02/03）本身有矛盾、或某约束在此场景下明显有害（如严格分层导致某合理需求无法实现）。

**处置**：**不得私自绕过约束**。约束是地基，绕过会让整个体系失效（风险 R6）。必须：
1. 停下，记录冲突的具体表现
2. 升级给人，附"遵守约束的代价" vs "修订约束的代价"
3. 人裁决 → 若修订约束，走 **ADR** 正式记录
4. ADR 落定后才继续

### D 类 — 范围/需求问题（必须升级）

发现要做的东西超出 plan 登记范围，或需求本身模糊/可能是 XY 问题。

**处置**：
1. **不自行扩大范围**（防风险 R5——造成 Agent OS 而非解决交接）
2. 对照 master-plan 第7节"范围纪律三问"自查
3. 升级给人确认：这个是否该做？是否偏离北极星（开发→QA 可靠交接闭环）？
4. 确认后才登记进 plan 并实施

## 4. 自主决策的授权边界

AI **可以自主**决定（无需升级）：

- A 类实现问题的技术方案
- 命名、代码组织、私有辅助函数的设计
- 在契约不变前提下的内部实现重构
- 仅影响本模块、不违约束、不改契约的模块 plan 微调（留痕）
- 测试用例的具体写法（满足必测场景前提下）

AI **必须升级**（不得自主）：

| 触发 | 为什么 |
|------|-------|
| C 类约束冲突 | 约束是地基，改它影响全局 |
| D 类范围扩张 | 防范围蔓延(R5) |
| 修改契约（05） | 影响多模块，破坏性 |
| 删除/跳过 plan 要求的功能 | 偏离目标 |
| 降低 DoD/覆盖率标准 | 放水即失控 |
| 两次根因分析仍无解 | 可能需要人的领域知识 |
| 任何破坏性/不可逆操作 | 安全 |
| 发现可能的 XY 问题 | 需求层面，人决策 |

## 5. plan 修正流程（B 类留痕）

AI 自主修订模块 plan 时，必须在该 plan 文件的"变更记录"区追加：

```markdown
## 变更记录
| 日期 | 变更 | 原因 | 影响 | 决策者 |
|------|------|------|------|-------|
| YYYY-MM-DD | 状态机增加 Cancelling 状态 | 实现发现用户取消需独立状态，原设计漏 | 仅本模块状态机+测试 | 实施AI自主 |
```

涉及契约或跨模块的，先升级，人确认后由 ADR 记录，再回填模块 plan。

## 6. 升级机制（如何向人求助）

升级不是"卡住了求救"，而是**带着分析和选项求裁决**。升级时必须提供：

```
┌─────────────────────────────────────────────────────────────┐
│                      合格的升级请求                          │
├─────────────────────────────────────────────────────────────┤
│ 1. 问题分类 (A/B/C/D) 与一句话概述                          │
│ 2. 上下文：在哪个模块、哪一步遇到                           │
│ 3. 已尝试什么、结果如何（证明不是没努力）                   │
│ 4. 根因分析：为什么会这样                                   │
│ 5. 选项：至少 2 个方案 + 各自代价/影响                       │
│ 6. 倾向建议 + 理由                                          │
│ 7. 阻塞范围：升级期间能否做别的模块                         │
└─────────────────────────────────────────────────────────────┘
```

❌ 不合格升级："这里报错了怎么办" / "设计好像有问题" / 不给选项的开放式提问。

## 7. 防失控的硬规则（应对 R6：AI 偏离且无人察觉）

| 规则 | 说明 |
|------|------|
| 不许声称未验证的完成 | DoD 未满足却标 ✅ = 严重违规 |
| 不许静默绕过约束 | C 类必须升级，绕过即埋雷 |
| 不许空测试凑数 | 无意义断言 = 测试质量红线 |
| 不许隐藏遗留 | 所有 TODO/债务登记到任务或风险 |
| 复审不信自述 | 关键模块 reviewer 独立验证证据 |
| 留痕一切偏离 | 任何对 plan 的偏离都要可追溯 |
| 北极星对齐 | 每里程碑回答"是否更接近开发→QA 可靠交接闭环" |

## 8. 与记忆系统的衔接（经验回写）

AI 在执行中积累的经验应回写到计划与约束，避免重复踩坑：

- 反复踩的坑 → 沉淀为"实施注意事项"，回写相关 goal / plan
- 某约束反复造成摩擦 → 作为 C 类信号，提请评估约束合理性
- 有效的实现模式 → 可提炼为项目级约束或 spec 补充

> 注：这里的"沉淀"指 AI 工程经验回写计划，与岗位能力沉淀无关；当前产品方向不含平台自学习。

## 9. 协议自检（每个模块完成时回答）

- [ ] 是否按标准工作循环执行（测试先行、门禁、自审、审计）？
- [ ] 遇到的问题是否正确分类并按类处置？
- [ ] 是否有未升级就自主做了"必须升级"的事？
- [ ] 所有对 plan 的偏离是否留痕？
- [ ] 是否有声称完成但 DoD 未满足的情况？
- [ ] 本模块是否让项目更接近北极星（开发→QA 可靠交接闭环）？

## 10. Definition of Done（三级，合并自原 plan/06）

"做完了没有"的唯一标准。AI 不得在未满足时标记完成。模糊的"差不多了"不算完成。粒度分三层递进：**任务级 → 模块级 → 里程碑级**。

### 10.1 任务级 DoD（每次提交）

- [ ] 实现了任务描述全部要求，主路径无 TODO 占位
- [ ] **先写的测试**现在通过（红→绿）
- [ ] 新增/修改逻辑有对应测试，断言有意义
- [ ] 编译零错误、静态检查零警告（clippy/eslint）、已格式化
- [ ] 无裸 `unwrap()`、无吞错误、无密钥明文、无调试残留
- [ ] 提交信息说明"做了什么 + 为什么"（中文）
- [ ] 不破坏已有测试

### 10.2 模块级 DoD（一个 Mxx 完成）

除满足任务级外：

**功能与契约**
- [ ] 模块 plan 列出的所有职责已实现
- [ ] 契约（trait+数据结构+错误）与 [ModuleContractSpec](../spec/ModuleContractSpec.md) 一致
- [ ] 涉及的状态机按 [StateMachineSpec](../spec/StateMachineSpec.md) 实现并测试

**宪法合规**（对照 [spec/](../spec/)）
- [ ] ProgramSpec 三原则、DDDSpec 领域建模
- [ ] EngineeringPrinciples SOLID 五条 + 可插拔（每条有证据）
- [ ] ArchitectureConstraints 依赖方向/边界/无循环
- [ ] 错误分层转换

**测试**（对照 [TestStrategy](../spec/TestStrategy.md)）
- [ ] 覆盖率达模块类型地板线
- [ ] 必测场景全覆盖（正常/边界/错误/状态机/并发/降级/契约）
- [ ] 无测试质量红线

**文档与审计**
- [ ] 模块 plan 更新为最终状态（含转换表、契约）
- [ ] 公开接口有文档注释
- [ ] 审计报告（templates/audit-report），checklist 每条带证据+证据等级
- [ ] 关键模块通过独立 reviewer 复审
- [ ] master-plan 看板更新为 ✅

**集成**
- [ ] 与依赖模块集成测试通过

### 10.3 里程碑级 DoD（一个 MS 切片完成）

- [ ] 该 MS 所有模块达模块级 DoD
- [ ] **端到端 demo 可复现**（有脚本/步骤跑通主链路）
- [ ] E2E 测试覆盖该切片主链路
- [ ] 该 MS 的"可验证产出"实际达成（见 [00-master-plan](./00-master-plan.md) 第4节）
- [ ] 范围纪律自查通过（master-plan 第7节三问）
- [ ] 里程碑审计报告（汇总各模块 + E2E）
- [ ] PLAN_LEDGER 全 `DONE`/`SUPERSEDED`，无未解决 `BLOCKED`/`CHANGED`
- [ ] 已知遗留登记，无隐藏债务

### 10.4 "未完成"判定（反清单）

出现任一，一律判未完成，不得标 ✅：

| 情况 | 为何不算完成 |
|------|------------|
| 测试失败或被 ignore 绕过 | 正确性未证明 |
| 覆盖率未达线 | 验证不充分 |
| 主路径有 TODO/未实现分支 | 功能不完整 |
| "本地能跑但没写测试" | 不可回归 |
| 审计报告缺失或打勾无证据 | 不可追溯 |
| 违反宪法但"先这样" | 技术债无声累积 |
| demo 跑不通但"逻辑应该对" | 端到端未验证 |
| 偏离 plan 且未走修正协议 | 失控 |
| 报告只有 SCENARIO_REGISTERED 却称"已验证" | 违反证据等级 |

### 10.5 速查卡

```
模块完成 = 功能全 + 契约合规 + 宪法合规(spec/) + 测试达标
         + 审计报告(带证据+等级) + 看板更新 + 集成通过
任一缺失 = 未完成。
```
