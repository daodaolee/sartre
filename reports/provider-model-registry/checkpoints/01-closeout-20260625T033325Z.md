# Provider Model Registry Closeout

## Scope

Implemented the first provider/model/executor registry slice for Sartre:

- Shared contracts for provider model profiles, capabilities, health reports, endpoint-scoped registry list, and selection resolution.
- SDK helpers for profile registration, health report, registry list, and selection resolve.
- Hub API bounded context with Postgres persistence and HTTP endpoints.
- Web Console read-only "模型" capability page scoped to the current endpoint identity.

Out of scope by design: real Codex/Claude execution, API key storage, provider gateway routing, chat composer, streaming output, quota, billing, and admin policy.

## Remaining Limitations

- Registry only records provider/model/executor facts and latest health. It does not run Codex, Claude, or any provider.
- Web Console is read-only for registry profiles. It intentionally has no API key editing, execution button, chat input, or streaming transcript.
- Selection resolve is deterministic and read-only; it does not account for quota, billing, tenant admin policy, or provider gateway routing.
- `ConversationLedger`, `ContextProjection`, and `ModelRun` are not changed in this slice; future work can link model runs to `provider_model_profile_id` if needed.
- Task attachment UI still projects storage URLs only; real COS/OSS upload belongs to a separate storage-provider goal.

## Next Goal Candidates

- Local Codex/Claude executor bridge: consume registry profile, validate command health, and create audited model runs without storing provider secrets.
- Provider gateway and secret management: dedicated bounded context for credential references, policy, and secure runtime injection.
- Streaming composer UI: conversation page that can call an executor through Hub while preserving provider-neutral ledger history.
- Registry-to-chat-runtime linking: optional `provider_model_profile_id` reference on model runs or context projections after executor behavior is stable.
